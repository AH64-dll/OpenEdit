```

kdenlive:duration           (e.g. "00:00:10.000")
kdenlive:file_hash          (md5 of the source file)
kdenlive:file_size          (bytes)
kdenlive:folderid           ("-1" for top-level bin)

kdenlive:docproperties.uuid (on a higher-level property)
kdenlive:docproperties.author / .modified / .created / .version
```

Plus a *huge* set of `meta.media.*` properties (width, height, fps,
codec, etc.) that `analyze`/`ffprobe` already produces — meaning
`mlt-pipeline` could populate most of these *for free* if its Go code
included them in the producer.

## Minimum checklist to stop being "Untitled"

For Phase 2 to emit a file Kdenlive opens as a real project (not
"Untitled"), the producer for each clip must include at least:

1. **`kdenlive:clipname`** — the human-readable name shown in the bin.
2. **`kdenlive:duration`** — must match the clip's real duration; Kdenlive uses it for bin display.
3. **`kdenlive:file_hash`** — md5 of the source file, used for "has the source changed?" detection.
4. **`kdenlive:file_size`** — bytes, paired with the hash.
5. **`kdenlive:folderid`** — `-1` for the top-level bin (no subfolders).
6. **`kdenlive:binType`** — `std` for AV clips.
7. **`kdenlive:clip_type`** — `0` for AV, `1` for image, `2` for audio, etc.

And on the project root (`<mlt>`):

8. **`kdenlive:docproperties.uuid`** somewhere (Kdenlive adds it as
   `<property name="kdenlive:docproperties.uuid">...</property>` on the
   *root* producer — i.e. on `<producer id="main_bin"/>`).
9. **`kdenlive:docproperties.version`** = `1.1` (or whatever the
   current Kdenlive writes).
10. **`kdenlive:docproperties.author`**, `.modified`, `.created` —
    optional but make the project's "Properties" dialog look real.

The profile itself does **not** need any `kdenlive:` properties; the
"Untitled" issue is on the *bin/producer* metadata, not the profile.

## What this means for Phase 2

Phase 2's project engine should:

1. Take the bare `<mlt>` the Go pipeline produces (or its own MLT
   output) and **add the `kdenlive:` properties on top**, by reading
   the source files (for `file_hash`, `file_size`) and the project
   state (for `clipname`, `folderid`, `docproperties.uuid`).
2. Generate a fresh `kdenlive:docproperties.uuid` if the project is
   new; reuse the existing one on round-trip edits.
3. Not touch the existing `mlt-pipeline` Go code; the Python engine
   reads what Go produced, decorates it, writes the `.kdenlive`.

This means `mlt-pipeline`'s "opens as Untitled" limitation is fixable
entirely in Python, without touching the Go pipeline. That's the
**ideal split of responsibilities**: Go = batch MLT generation, Python
= Kdenlive-aware wrapping.
