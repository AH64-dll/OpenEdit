# v1.7 — LLM Provider Bridge (multi-CLI selection)

**Status:** draft (post-Phase-0 spike)
**Date:** 2026-07-22
**Author:** opencode (brainstorming session)

## 0. Problem

The Open Edit chat backend currently has a single LLM provider selection knob:
`OPEN_EDIT_LLM_PROVIDER` ∈ {`anthropic`, `openai`, `pi`}. The user wants to
switch between the CLIs they have installed (`pi`, `opencode`) and the model
catalogs those CLIs expose (e.g. `opencode-go/minimax-m3`,
`omniroute/antigravity/gemini-2.5-pro`) **per project**, **from the chat UI**,
without restarting the server.

A second goal is to wire in the `opencode-go` subscription as a first-class
provider (the existing `pi` provider already uses it as its `OPEN_EDIT_PI_PROVIDER`
default; the new `opencode` provider will reach the same subscription via the
`opencode-go` model prefix).

## 1. Decisions locked before design (from brainstorming review)

1. **Config shape** — provider enum is `{anthropic, openai, pi, opencode}`.
   `antigravity` is **not** a provider. It is a UI label that maps to
   `provider=opencode, model=omniroute/antigravity/<model>`.
2. **Config file** — per-project `.open_edit/config.toml`. Server reads it on
   every chat turn (env vars remain the default if the file is absent).
3. **Selection bar UX** — two dropdowns (Provider, Model) at the top of the
   chat panel. Model list depends on the chosen provider.
4. **Phase 0 first** — capture real `pi` and `opencode` JSON output, answer
   the three questions, then design the adapter interface.
5. **Adapter minimal, evidence-driven** — only the methods Phase 0 actually
   justified. No factories, no DI container.
6. **TDD** — failing tests first, then implementation.

## 2. Phase 0 — Spike findings

Captured samples:
- `pi --print --mode json --no-extensions --session-id oe-spike-pi-1 "say hello in 5 words"` → `/tmp/pi_sample.jsonl`
- `opencode run --format json "say hello in 5 words"` → `/tmp/opencode_sample.jsonl`
- `opencode run --format json -s <sid> "and goodbye"` → `/tmp/opencode_resume.jsonl` (session resume)
- `opencode providers list` → confirms opencode manages its own auth

### Q1 — Are the JSON shapes close enough to share a parser?

**Answer: No.** They are different event vocabularies:

| pi                                                      | opencode                                |
| ------------------------------------------------------- | --------------------------------------- |
| `{"type":"session","version":3,"id":"...","timestamp":...}` | `{"type":"step_start",...,"sessionID":"ses_..."}` |
| `{"type":"agent_start"}` / `{"type":"agent_end",...}`   | implicit (first/last `step_*`)          |
| `{"type":"turn_start"}` / `{"type":"turn_end","message":{...},"toolResults":[...]}` | `{"type":"step_finish","part":{...,"reason":"stop","tokens":{...},"cost":0}}` |
| `{"type":"message_start","message":{"role":"assistant","api":"anthropic-messages","provider":"opencode-go","model":"minimax-m3",...}}` | `{"type":"text","timestamp":...,"sessionID":"...","part":{...,"text":"Hello!..."}}` |
| `{"type":"message_end","message":{...,"usage":{...,"cost":{...}}}` | `{"type":"step_finish","part":{...,"tokens":{...,"cache":{...}},"cost":0}}` |
| Tool events: `toolCall` / `toolResult` blocks nested in `message.content[]` | Tool events not in `run` output for non-agent mode; `run` is chat-first, tools are agent-mode |
| Session resumption via `--session-id <oe-...>`            | Session resumption via `-s ses_...` (different id format) |

**Decision:** keep the two adapters independent. Share only the `ChatEvent`
output shape (the existing `StreamEvent` TypedDict in `llm.py`) and the
envelope. The "adapter" interface is a thin facade, not a shared parser.

### Q2 — Does opencode support session resume, and what does it cost?

**Answer: Yes, but with a twist.** `opencode run -s <sessionID> "followup"`
resumes the same session. The captured resume sample shows:
- New `step_start` / `text` / `step_finish` event triple for the new turn
- The new turn's `tokens.input` = **60** (the new message only), but
  `tokens.cache.read` = **12672** (the prior context is read from cache)

**Cost implication:** opencode reuses cached context, so cost-per-turn does
NOT grow linearly with conversation length. Same cost shape as pi. **No
special UI handling needed.** The existing `usage` event with
`source="computed"` semantics will work for opencode as-is.

(Verified: `cost: 0` in the resume sample because the chosen default model
returns zero cost in this user's pricing; for paid models the same shape
will produce nonzero values. Confirmed by the structure of the
`step_finish.part.cost` field.)

### Q3 — Does opencode want an API key per invocation, or does it manage its own?

**Answer: opencode manages its own auth.** `opencode providers list` shows
`OpenCode Go` credential is stored at `~/.local/share/opencode/auth.json`
(no env var required). Same for the user's `omniroute/antigravity/*` models
(also authenticated via that file). **No `OPENCODE_GO_API_KEY` env var wiring
is needed in Open Edit** — the existing `OPEN_EDIT_LLM_API_KEY` remains
unused for the opencode path; that env var only matters for the `anthropic`
and `openai` direct-SDK providers.

**Decision:** the opencode adapter does NOT pass an API key. It just runs
the CLI. This is `manages_own_auth() == True` in the adapter interface.

### Q4 — model naming divergence (bonus finding)

The user's `~/.config/opencode/opencode.json` lists `antigravity/gemini-2.5-pro`
as a model name (two-segment). The actual `opencode models` output shows
`omniroute/antigravity/gemini-2.5-pro` (three-segment). The two-segment form
is rejected at runtime (`{"type":"error",...,"UnknownError":...}`) and the
three-segment form hung in the spike (likely a server-side issue with the
`omniroute` provider; the spike was done on a model that has a known
weekly-limit error so we couldn't get a clean transcript).

**Decision:** the config schema requires the **three-segment** form for
antigravity models: `omniroute/antigravity/<model>`. The UI's "Antigravity"
preset button writes that exact string. If the user types a different form
in the Model field, we pass it through verbatim — the opencode CLI is the
source of truth for what model names it accepts.

## 3. Adapter interface (minimal, evidence-driven)

```python
class CLIAdapter:
    """One CLI backend (pi or opencode). Stateless; methods only."""

    name: str  # "pi" | "opencode"

    def build_command(
        self,
        model: str,
        user_text: str,
        session_id: str,
        extension_path: str | None,
        system_prompt: str,
    ) -> list[str]:
        """Return the full argv to exec. No shell quoting needed."""
        ...

    def parse_events(self, stdout_bytes: AsyncIterator[bytes]) -> AsyncIterator[StreamEvent]:
        """Read raw stdout lines, yield our ChatEvent dicts.

        For pi: line-delimited JSON, then run through _pi_normalize_event.
        For opencode: line-delimited JSON, then run through
        _opencode_normalize_event (new in this design).
        """
        ...

    def supports_tools(self) -> bool:
        """True iff the CLI has a tool-extension mechanism that Open Edit
        can wire its 11 tools into. Currently only pi (extension.ts)."""
        ...

    def supports_images(self) -> bool:
        """True iff the adapter can forward image inputs to the model.
        Pi: yes (via toolResult image blocks). Opencode: yes (via
        --file attachments, but only when not in 'run' mode — for
        the chat adapter we treat it as False for v1.7)."""
        ...

    def manages_own_auth(self) -> bool:
        """True iff the CLI reads auth from its own local config and
        Open Edit does not need to pass a key. Opencode: True.
        Pi: False (pi can be told --provider opencode-go, which uses
        OpenCode Go's subscription via the same auth file, but pi
        itself does not auto-load that file; it expects either
        OPENCODE_GO_API_KEY in env or a logged-in session)."""
        ...

    def default_model(self) -> str:
        """Sensible default for the model dropdown when this adapter
        is selected. 'minimax-m3' for pi, 'opencode-go/minimax-m3' for
        opencode (same model, exposed via the opencode-go prefix)."""
        ...

    def available_models(self) -> list[str]:
        """Static list of model ids this adapter supports. For pi we
        hand-curate (no easy introspection). For opencode we shell out
        to `opencode models` and cache the result for 60s."""
        ...
```

**Why each method exists** — every method above is justified by a real need:
- `build_command` — adapters differ on flag syntax (pi uses `--print`,
  `--mode json`, `--append-system-prompt`; opencode uses positional message
  + `--format json` + `--title` for session).
- `parse_events` — adapters differ on event vocabulary (Q1).
- `supports_tools` — the existing 11 Open Edit tools are wired via
  `pi_extension/extension.ts` which is pi-specific. Opencode has no
  equivalent yet. The UI must know which providers can trigger tools.
- `manages_own_auth` — opencode reads its own auth file. Pi does not.
  Knowing this prevents the agent from spuriously trying to wire
  `OPENCODE_GO_API_KEY` (which opencode does not need).
- `default_model` — UI preset button.
- `available_models` — populates the Model dropdown. The list comes from
  `opencode models` (cached) for opencode and a hardcoded list for pi.

## 4. File-level changes

### `open_edit/serve/llm.py`
- Extract `_stream_pi` into a generic `_stream_cli(adapter, model, ...)`.
- `PiAdapter` wraps the existing pi-specific build/parse code verbatim
  (cut-paste from the current `_stream_pi` and `_pi_normalize_event`).
- Add `OpenCodeAdapter` (new in this design).
- `stream_chat` dispatch: `if provider in {"pi", "opencode"}: yield from
  _stream_cli(adapter_for(provider), ...)`.
- Add `_resolve_adapter(provider: str) -> CLIAdapter` — a dict lookup, no
  factory, no DI container.

### `open_edit/serve/opencode_adapter.py` (new)
- `OpenCodeAdapter` class.
- `_opencode_normalize_event(obj)` — maps opencode's `step_start` /
  `text` / `step_finish` / `error` events to our `StreamEvent` shape.
- `_opencode_available_models()` — shells out to `opencode models` (cached
  60s via simple module-level lru_cache).
- ~150 lines new code.

### `open_edit/serve/llm_config.py` (new)
- Pydantic models for the `.open_edit/config.toml` schema:
  - `LLMConfig` with `provider: Literal["anthropic","openai","pi","opencode"]`,
    `model: str`, and a `cli: dict[str, str]` section for adapter-specific
    overrides (currently empty; reserved).
- `load_llm_config(project_dir: Path) -> LLMConfig` — reads
  `<project_dir>/.open_edit/config.toml`. Falls back to env vars
  (`OPEN_EDIT_LLM_PROVIDER`, `OPEN_EDIT_LLM_MODEL`) if the file is missing
  or invalid (with a single stderr warning).
- ~60 lines new code.

### `open_edit/serve/app.py`
- Add `GET /api/projects/{project_id}/llm-config` → returns current
  config (provider, model, available_models for the current provider).
- Add `PUT /api/projects/{project_id}/llm-config` body=`{provider,
  model}` → writes to `.open_edit/config.toml`. Validates that
  `provider` is in the enum; if `provider == "opencode"`, validates that
  `model` is non-empty. Returns the saved config.
- ~50 lines new code.

### `open_edit/serve/static/index.html` + `app.js`
- Add the selection bar to `.topbar-center`:
  `<label>Provider:</label> <select id="provider-select"> ... </select>`
  `<label>Model:</label> <select id="model-select"> ... </select>`
- On change: PUT to `/api/projects/{id}/llm-config`, then trigger a
  reconnect of the WebSocket so the new provider applies to the next
  message.
- If `provider.supports_tools() == false`, grey out the chat input or
  show a banner "Tool calls are unavailable with this provider".
- ~80 lines new JS + ~10 lines HTML.

### `open_edit/serve/pi_extension/extension.ts`
- **Unchanged.** Note in this spec that the TS extension stays pi-only.
  The opencode adapter has `supports_tools() == False` until an
  equivalent opencode extension is built (out of scope for v1.7).

## 6. Acceptance criteria

- **A1** — selecting "OpenCode" in the bar sets `provider=opencode,
  model=opencode-go/minimax-m3` in `.open_edit/config.toml`, the next
  chat message is dispatched to the opencode CLI, and the response streams
  back to the UI.
- **A2** — selecting "Antigravity" sets `provider=opencode, model=
  omniroute/antigravity/gemini-2.5-flash` (or whatever the user picks).
  The server never sees the string "antigravity" as a provider; it only
  appears as a model-name prefix.
- **A3** — `OPEN_EDIT_LLM_PROVIDER=antigravity` is rejected at startup
  with a clear error (the enum does not include `antigravity`).
- **A4** — The pi provider still works identically (Phase 1 acceptance
  from the review: no user-visible diff for the existing pi path).
- **A5** — Tool-triggering UI is hidden/disabled when
  `supports_tools() == False` (i.e. when provider is `opencode`).
- **A6** — All new env-var wiring is documented in the module docstring
  of `llm.py`. No silent reading of new env vars without docs.
- **A7** — Full test suite passes (`pytest open_edit/tests/`); new tests
  are at least 80% of the lines they cover.
- **A8** — Ruff + mypy pass on new files.

## 7. Out of scope for v1.7

- Building an opencode-side tool extension to register the 11 Open Edit
  tools with the opencode CLI. When (and if) this is built, the
  `OpenCodeAdapter.supports_tools()` flag flips to True and the UI
  unhides the tools.
- Cost math for opencode — we read `step_finish.part.cost` and
  `step_finish.part.tokens` directly; no separate `cost.py` rule is
  needed because the adapter emits a `usage` event with the values.
- Per-user-global config fallback (e.g. `~/OpenEditProjects/_global.toml`).
  Config is per-project only for v1.7.
- The pi provider's existing session-file delta-based cost computation
  (`cost_mod.parse_pi_session_usage_delta`) is reused as-is. No
  opencode-equivalent cost module is needed because the opencode adapter
  reports cost directly in the `step_finish` event.

## 8. Risks and mitigations

- **R1** — opencode CLI surface may change. Mitigation: version-pin
  the opencode binary or document the tested version in the adapter
  docstring. `opencode --version` is logged on adapter init.
- **R2** — `opencode run` may be deprecated in favor of `opencode
  attach` + a running server. Mitigation: the adapter is small enough
  to rewrite the `build_command` method in <30 lines if needed.
- **R3** — The user's `opencode-go` weekly limit is currently exhausted
  (per the Phase 0 spike). This is unrelated to the design but worth
  noting — the user cannot currently end-to-end test the opencode
  provider against a real model. Tests will use a fake opencode CLI.
- **R4** — The three-segment `omniroute/antigravity/*` form hung during
  the spike. This is an opencode-server-side issue (likely a known bug
  in the omniroute provider for this user's account). The adapter
  passes the model name verbatim; the hang surfaces as a streaming
  timeout in the UI, which is acceptable behavior.
