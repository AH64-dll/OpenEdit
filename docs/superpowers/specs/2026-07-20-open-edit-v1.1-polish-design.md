# Open Edit v1.1 — Polish Design

**Status:** Draft (brainstorming complete; pending user review)
**Date:** 2026-07-20
**Supersedes:** n/a
**Builds on:** `2026-07-20-open-edit-design.md` (v1), `2026-07-22-phase4-design-audit-fixes.md`, final whole-branch review at commit `5ceb596`
**Outcome target:** v1.1 of Open Edit is the same product surface as v1, but every review-flagged quality item from the Phase 4 + 4.5 final review is resolved. No new features.

## 1. Goal

Bring Open Edit v1 to release-quality by resolving the **7 review-flagged polish items** that the final whole-branch review of Phase 4 + 4.5 left as `M` (minor) or deferred `I` (important). After v1.1:

- All 5 Python e2e tests pass in a runnable environment (conftest fixture fixed).
- All 6 spec-listed Rust integration tests are implemented (5 currently missing — 1 conditional on env).
- The render sandbox's documented cgroup limits actually apply in production (setup script provided).
- The free-form sandbox binary's stdout is clean of child-process pollution.
- The bootstrap-safety-net regression test catches future ops added to `OperationUnion` without updating the sandbox's `op_types` list.
- No unused imports, no dead flags, no silently-swallowed errors.
- A test asserts the hard-cap clamp at the wrapper level (currently tested only at the constant level).

**Non-goal (v1.1):** No new user-visible features. No Tauri, no multi-user, no model integrations, no bespoke motion graphics. Those are v2.

## 2. Scope (the 7 tasks)

Each task is a separate commit. Each is TDD where applicable. Each has its own per-task review.

### T1 — Conftest fixture fix (resolves FR I-1)

**File:** `open_edit/tests/conftest.py:11-31` (`tmp_project_with_assets`)

The fixture currently constructs a `Project` whose `assets` dict is in-memory only. `run_free_form` takes a `workdir` and loads assets from the on-disk edit graph, not from the in-memory `Project.assets`. The 5 e2e tests in `test_free_form_e2e.py` use this fixture and then call `run_free_form`, which means in a runnable environment, the asset on disk is never loaded, the first `ir.add_clip(asset_hash='abc123', ...)` call fails with `asset_hash 'abc123' not in project`, and 4 of 5 e2e tests fail.

**Fix (per FR recommended next step, option a):**
1. Use `AssetStore.ingest_paths()` to write the asset into the fixture's CAS layout.
2. Seed the edit graph with the `AddClipOp` that references that asset.
3. Remove the in-memory `assets` dict from the returned `Project` (it was misleading).

**Acceptance:** `python -m pytest open_edit/tests/test_free_form_e2e.py -v` runs all 5 e2e tests against a real (or `bwrap`/cgroup-stubbed) sandbox. The 4 currently-failing tests now pass; the 1 that was always passing stays passing.

**Out of scope:** Re-enabling the 5 e2e tests on the CI without bwrap. The e2e tests stay skipped in environments without bwrap; this fix unblocks them only in bwrap-capable environments.

### T2 — Render sandbox cgroup setup script

**File:** `bin/setup_render_cgroup.sh` (new), `open_edit/sandbox/src/render_jail.rs:6-12` (constant), `docs/operator-setup.md` (new)

`apply_cgroup_limits` writes to `/sys/fs/cgroup/open_edit_render/{memory.max,cpu.max,cgroup.procs}`. The directory must exist; if it doesn't, I3 (already fixed) now returns an error instead of silently no-opping. Operators need a one-liner to create the directory and grant the current user write access.

**Fix:**
1. New `bin/setup_render_cgroup.sh` (bash, ~20 lines, idempotent) that:
   - Verifies cgroup v2 is mounted (`stat -fc %T /sys/fs/cgroup` = `cgroup2fs`).
   - Creates `/sys/fs/cgroup/open_edit_render` if missing.
   - Enables `memory` and `cpu` controllers (writes `+memory +cpu` to `cgroup.subtree_control`).
   - Prints clear instructions if cgroup v2 is not available.
2. New `docs/operator-setup.md` (or extend existing README) that documents the one-liner.
3. Add a test that the script's help output mentions cgroup v2 and the directory path (shell-script smoke test using `bash -n` for syntax and `--help` for output).

**Acceptance:** Script exists, syntax-checks, prints clear instructions, and the doc explains when/why to run it.

### T3 — 5 missing Rust integration tests (resolves FR M-4)

**File:** `open_edit/sandbox/tests/integration.rs` (extend)

The spec §7.1 lists 6 tests. Current state: only `e2e_python_runs_and_writes_ops` (ignored, partially written) is from the spec list. `bwrap_unavailable_reason` exists but is not in the spec list. **5 of 6 spec tests are missing.**

**Fix (write 5; the 5th is conditional on env):**
1. `e2e_network_blocked` — script that calls `urllib.request.urlopen('http://example.com')` exits non-zero with stderr mentioning network; assert exit code != 0 and stderr contains "Network" or "denied" or similar.
2. `e2e_source_ro_blocks_writes` — script that tries to write to a read-only-mounted source file exits non-zero; assert exit code != 0 and stderr mentions "read-only" or "Permission".
3. `e2e_timeout_kills_runaway` — script that calls `time.sleep(60)` gets killed at the 5s timeout; assert exit code != 0 and stderr contains "timeout".
4. `e2e_parent_id_stamped` — script that calls `ir.add_clip(...)` produces a JSONL with `parent_id` matching the `parent_op_id` argument; assert the JSONL line has the correct `parent_id`.
5. `e2e_python_version_mismatch` — script run with `--expected-py-version 3.11` but a Python 3.14 binary exits non-zero with stderr mentioning version mismatch. **Conditional:** only write if the dev env has BOTH `python3.11` AND `python3.14` available; otherwise the test body is a no-op `#[ignore]`-d stub with a TODO comment.

**Acceptance:** `cargo test --features integration` enumerates 6 spec-listed tests + 1 spec-out-of-scope (`bwrap_unavailable_reason`) = 7 tests. All 6 spec-listed tests exist (5 new + 1 ignored existing); 5 are `#[ignore]`-d (require bwrap + cgroup v2); 1 (`bwrap_unavailable_reason`) runs without bwrap.

### T4 — Sandbox binary stdout pollution (resolves FR M-1)

**File:** `open_edit/sandbox/src/main.rs:74-77`

`std::process::Command::new("bwrap")` is spawned with `.spawn()` which inherits the parent's stdout. If the free-form script calls `print("hello")`, that output mixes with the JSON line on stdout. The Python wrapper's `json.loads(proc.stdout)` then fails with `sandbox_protocol_error`.

**Fix:**
1. Change `cmd.spawn()` to `cmd.stdout(Stdio::piped()).stderr(Stdio::piped())`.
2. In the bwrap result handler, read `child.stdout` and `child.stderr` into separate buffers.
3. Discard the captured `child.stdout` (it's the script's user `print()` calls; the JSON is on the wrapper's stdout, not the child's).
4. Keep `child.stderr` for the `Output.stderr` field (current code already populates it via `String::new()` — replace with the actual captured stderr).

**Acceptance:** New test: a free-form script that does `print("hello"); ir.add_clip(...)` produces valid JSON on the wrapper's stdout, and `Output.stderr` contains the captured script stderr (e.g., from `import warnings; warnings.warn("x")`).

### T5 — Hard-cap clamp direct test (resolves FR M-3)

**File:** `open_edit/tests/test_sandbox_bridge.py:108-117`

The test currently only asserts the constants exist with the right values:
```python
assert MAX_FREEFORM_TIMEOUT_SEC == 300
assert MAX_FREEFORM_MEM_MB == 4096
```

The clamping logic at `sandbox_bridge.py:103-104` (`timeout = min(int(timeout), MAX_FREEFORM_TIMEOUT_SEC)`) is correct but untested.

**Fix:** Add 2 tests:
1. `test_timeout_clamped_to_max` — call `run_free_form(workdir, code, timeout=999_999)` (or the underlying wrapper); assert the value passed to the Rust binary is `MAX_FREEFORM_TIMEOUT_SEC` (e.g., via a mock on the Rust invocation).
2. `test_mem_mb_clamped_to_max` — same for memory.

(These tests can mock the Rust invocation so they run without bwrap.)

**Acceptance:** The clamp behavior is directly tested; a future refactor that breaks the clamp will be caught.

**Files:**
- `open_edit/tests/test_sandbox_bridge.py:13-21` — 6 unused imports (M-7)
- `open_edit/sandbox/src/main.rs:47-48` — `--ops-output` flag is parsed but never used (M-6)
- `open_edit/sandbox/tests/integration.rs:19` — ignored test's `_bootstrap.py` would fail in the sandbox (M-5)

**Fixes:**

2. **M-6:** Either (a) remove the `--ops-output` flag from `Cli` and the wrapper, or (b) wire it up. Pick (a) — it's dead; remove it.
3. **M-5:** Update the ignored test's `_bootstrap.py` to be self-contained (inline the IR/op models the way the actual `sandbox_bridge._render_bootstrap` does). This is a "test-quality" fix; the test will still be `#[ignore]`-d, but if/when it's un-ignored, it will work.

**Acceptance:** No unused imports; no dead CLI flags; the ignored test's bootstrap is self-contained.

### T7 — IR bootstrap safety net (resolves FR C1 follow-up)

**File:** `open_edit/open_edit/agent/sandbox_bridge.py:339-345` (`_render_bootstrap`), `open_edit/tests/test_sandbox_bridge.py` (existing C1 test)

The C1 fix (already in `5ceb596`) hardcoded the expected list of 24 op names. If a future op is added to `OperationUnion` in `ir/types.py` without updating the `op_types` list in `sandbox_bridge.py`, the regression test only catches it if the test author remembers to update the `expected` list.

**Fix:**
1. Replace the hardcoded `expected` list in `test_render_bootstrap_inlines_all_24_op_classes` with a list derived from `OperationUnion` itself (using `pydantic.TypeAdapter(OperationUnion).json_schema()['oneOf']` to enumerate member types).
2. The test then asserts: every member of `OperationUnion` has a class definition in the bootstrap.

**Acceptance:** Adding a 25th op class to `OperationUnion` causes the test to fail with a clear "missing op class definitions: [<new op>]" message. The C1 regression is fully self-enforcing.

## 5. Error handling

| Failure | Before v1.1 | After v1.1 |
|---|---|---|
| Cgroup dir missing | I3 (fixed in `5ceb596`) returns error | Operator runs `bin/setup_render_cgroup.sh`; error becomes informative ("cgroup dir missing — run setup script") |
| Free-form script `print()` | `sandbox_protocol_error` on `json.loads` | JSON parse succeeds; `print()` output is silently consumed (or routed to logs at DEBUG level) |
| LLM calls an IR method added after v1.1 | Silently broken (bootstrap missing the class) | Regression test fails in CI with the specific missing class name |
| Future op added to `OperationUnion` | C1 regression test passes if author updates `expected`; silently broken if not | Self-enforcing test that always catches it |
| e2e tests run in bwrap env | 4 of 5 fail at first `ir.add_clip` | All 5 pass |
| Unused imports / dead flags | Cosmetic but noise | Removed |

## 9. Decision log

| Decision | Why |
|---|---|
| One plan, 7 focused tasks (vs. two plans) | The user picked "All review-flagged polish" + "One plan, 7 focused tasks" in brainstorming. Total scope is ~10h; splitting adds coordination cost without benefit. |
| T3 excludes `e2e_python_version_mismatch` if env lacks both Pythons | That test needs two Python interpreters on the host; the dev env may not have both. Test-the-env guard. |
| T6 picks (a) for M-6 (remove `--ops-output` flag) instead of (b) wire it up | The flag is dead code; the wrapper reconstructs the host path from `--scratch`. Removing is the right call. |
| T7 derives expected list from `OperationUnion` instead of keeping a hardcoded list | The whole point of "safety net" is to be self-enforcing. Hardcoding the expected list shifts the burden to future authors. |
| T1 doesn't try to make the 5 e2e tests run on the CI without bwrap | The e2e tests are inherently bwrap-dependent; unskipping them on CI is a separate (and out-of-scope) work item. |

## 10. Risks

| Risk | Mitigation |
|---|---|
| T1's conftest refactor breaks the 16 currently-passing tests that use `tmp_project_with_assets` | Run the full test suite after the refactor; if anything breaks, fix the fixture (don't roll back the I-1 fix). |
| T3's 5 new tests (4 unconditional + 1 conditional) fail in bwrap env for environmental reasons (missing /sys/fs/cgroup, etc.) | Tests are `#[ignore]`-d; they assert the test exists and is syntactically valid, not that they pass on the dev box. |
| T4's stdout pollution fix changes the sandbox's output format | The only consumer is `sandbox_bridge.run_free_form`; check it still parses the output. The change is "discard child stdout" which removes user `print()` calls — not JSON. |
| T6's removal of `--ops-output` flag breaks some unknown caller | `grep` for `--ops-output` first; if any caller exists, switch to option (b) and wire it up. |
