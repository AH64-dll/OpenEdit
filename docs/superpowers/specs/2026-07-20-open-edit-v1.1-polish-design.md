### T1 — Conftest fixture fix (resolves FR I-1)

**File:** `open_edit/tests/conftest.py:11-31` (`tmp_project_with_assets`)

The fixture currently constructs a `Project` whose `assets` dict is in-memory only. `run_free_form` takes a `workdir` and loads assets from the on-disk edit graph, not from the in-memory `Project.assets`. The 5 e2e tests in `test_free_form_e2e.py` use this fixture and then call `run_free_form`, which means in a runnable environment, the asset on disk is never loaded, the first `ir.add_clip(asset_hash='abc123', ...)` call fails with `asset_hash 'abc123' not in project`, and four of the five e2e tests fail.

**Fix (per FR recommended next step, option a):**

```python
@pytest.fixture
def tmp_project_with_assets(tmp_path):
    """A project with one asset pre-ingested, suitable for free-form runs (L9)."""
    from open_edit.ir.types import AddClipOp, Asset
    from open_edit.storage.edit_graph import EditGraphStore
    from open_edit.storage.assets import AssetStore

    # Step 1: ingest a real asset via AssetStore (writes to CAS layout)
    store = AssetStore(tmp_path / "assets")
    asset = store.ingest_paths([Path("/tmp/clip.mp4")])[0]
    # `asset.asset_hash` is now real (SHA-256 of the file content)

    # Step 2: seed the edit graph with an AddClipOp that references the asset
    graph = EditGraphStore(tmp_path / "edit_graph.db")
    seed_op = AddClipOp(
        author="user",
        asset_hash=asset.asset_hash,
        track_id="video_main",
        position_sec=0.0,
        in_point_sec=0.0,
        out_point_sec=asset.duration_sec,
    )
    graph.append(seed_op)  # writes to edit_graph.db; returns sequence_num

    # Step 3: return a Project without the misleading in-memory assets dict
    # (the sandbox loads from disk via _load_assets_via_store)
    return Project(name="test", workdir=tmp_path)
```

**Acceptance:** `python -m pytest open_edit/tests/test_free_form_e2e.py -v` runs all 5 e2e tests against a real (or `bwrap`/cgroup-stubbed) sandbox. Four previously-failing tests now pass; the one that was always passing stays passing.

**Out of scope:** Re-enabling the 5 e2e tests on the CI without bwrap. The e2e tests stay skipped in environments without bwrap; this fix unblocks them only in bwrap-capable environments.

### T2 — Render sandbox cgroup setup script

**File:** `bin/setup_render_cgroup.sh` (new), `open_edit/sandbox/src/render_jail.rs:6-12` (constant), `docs/operator-setup.md` (new)

`apply_cgroup_limits` writes to `/sys/fs/cgroup/open_edit_render/{memory.max,cpu.max,cgroup.procs}`. The directory must exist; if it doesn't, I3 (already fixed) now returns an error instead of silently no-opping. Operators need a one-liner to create the directory and grant the current user write access.

**Fix (cgroup v2 mechanics — corrected):**

cgroup v2 has two separate control files:

- `/sys/fs/cgroup/cgroup.subtree_control` (the **parent's** file) — controls which controllers are available to **child cgroups** under that parent. To make `memory` and `cpu` available inside `open_edit_render`, you write `+memory +cpu` to the **PARENT's** `cgroup.subtree_control` (i.e., `/sys/fs/cgroup/cgroup.subtree_control`), NOT to the child's own file.
- `/sys/fs/cgroup/open_edit_render/cgroup.subtree_control` (the child's file) — controls which controllers are available to the **child's children**, not to the child itself.

The script:

1. Verifies cgroup v2 is mounted (`stat -fc %T /sys/fs/cgroup` = `cgroup2fs`). Exits with clear message if not.
2. Creates `/sys/fs/cgroup/open_edit_render` (uses `mkdir -p` for idempotency).
3. Writes `+memory +cpu` to `/sys/fs/cgroup/cgroup.subtree_control` (the parent, to enable controllers for the child). Uses `grep -q` to check if controllers are already enabled before writing (avoids `EBUSY` on some kernels).
4. Handles the cgroup v2 "no internal processes" constraint: if a previous run left processes in the root cgroup, enabling new controllers may fail. The script detects this (`cat /sys/fs/cgroup/cgroup.events | grep populated`) and prints a workaround ("move processes to a child cgroup, or run on a fresh system").
5. Privilege model: writes are to root-owned files. Script requires `sudo` invocation. The privilege step `sudo chown -R "$USER" /sys/fs/cgroup/open_edit_render` (so subsequent unprivileged renders can write to `cgroup.procs`) is included. Note: this resets on reboot; the script is designed to be re-run after each boot.
6. Prints a clear "cgroup ready" message on success, or specific failure modes on error.

**Idempotency checklist:**
- `mkdir -p` (not bare `mkdir`)
- Check if controllers are already enabled before writing `+memory +cpu` to parent's `subtree_control`
- Check if cgroup already has the right ownership before `chown`
- Print warnings, not errors, for "already in desired state" cases
- `set -euo pipefail` at top of script
- `shellcheck bin/setup_render_cgroup.sh` clean (no warnings)

**New `docs/operator-setup.md`** documents:
- When to run the script (once per boot, or as a systemd unit)
- Why cgroups are needed (MemoryMax=4G, CPUQuota=300% documented in spec)
- How to verify (`cat /sys/fs/cgroup/open_edit_render/memory.max` should show `4294967296`)
- How to clean up (`sudo rmdir /sys/fs/cgroup/open_edit_render` after stopping all renders)

**Acceptance:** Script exists, passes `bash -n`, passes `shellcheck` with zero warnings, prints clear instructions, the doc explains when/why to run it, and the script is idempotent (running it twice in a row produces no errors).

### T4 — Sandbox binary stdout pollution (resolves FR M-1)

**File:** `open_edit/sandbox/src/main.rs:74-77`

**Data flow (corrected):**
```
Rust binary → spawns bwrap → bwrap runs Python script
  Python script's stdout (user print() calls + IR ops JSONL) → bwrap's stdout → Rust child.stdout
  Python script's stderr → bwrap's stderr → Rust child.stderr
Rust binary writes its protocol JSON ({"exit_code":..., "stderr":..., "duration_s":...}) to its OWN stdout
  → Python wrapper reads this from the Rust process's stdout
```

So `child.stdout` carries the script's user `print()` calls AND its IR ops JSONL; both are noise from the wrapper's perspective (ops are written to a file, not stdout). The protocol JSON is on the Rust process's own stdout (the `println!` at the end of `main()`), and the wrapper reads that. With the current code, `cmd.spawn()` inherits the parent's stdout — meaning `child.stdout == rust_process.stdout` — and the script's `print()` calls mix with the protocol JSON on the same line. The Python wrapper's `json.loads(proc.stdout)` then fails with `sandbox_protocol_error`.

**Fix:**
1. Change `cmd.spawn()` to `cmd.stdout(Stdio::piped()).stderr(Stdio::piped())`. This decouples `child.stdout` from the Rust process's stdout.
2. Use `child.wait_with_output()` (which reads both pipes concurrently via internal threads) to capture `child.stdout` and `child.stderr`. **Do NOT** use sequential `read_to_string(child.stdout)` then `read_to_string(child.stderr)` — if the script writes >64KB to stdout, the pipe buffer fills, the child blocks on `write()`, and the parent blocks on `wait()` → **deadlock**. `wait_with_output()` is deadlock-safe.
3. Discard the captured `child.stdout` (it's script noise — user `print()` calls and the IR ops JSONL, both of which are noise on the protocol channel).
4. Populate `Output.stderr` with the actual captured `child.stderr` (currently `String::new()` — replace with the captured buffer).
5. The Rust binary's own `println!` of the protocol JSON is unchanged — it goes to the Rust process's stdout, which the Python wrapper reads.

**Acceptance:** New test: a free-form script that does `print("hello"); ir.add_clip(...)` produces valid JSON on the wrapper's stdout, and `Output.stderr` contains the captured script stderr (e.g., from `import warnings; warnings.warn("x")`). The script's `print("hello")` is silently consumed (not visible in either the wrapper's parsed JSON or `Output.stderr`).

**Risk:** If `child.wait_with_output()` is not used (e.g., a future refactor uses manual pipe reads), the deadlock returns. Mitigation: add a comment in `main.rs` explaining the requirement.

### T5 — Hard-cap clamp direct test (resolves FR M-3)

**File:** `open_edit/tests/test_sandbox_bridge.py:108-117`

The test currently only asserts the constants exist with the right values:
```python
assert MAX_FREEFORM_TIMEOUT_SEC == 300
assert MAX_FREEFORM_MEM_MB == 4096
```

The clamping logic at `sandbox_bridge.py:103-104` (`timeout = min(int(timeout), MAX_FREEFORM_TIMEOUT_SEC)`) is correct but untested.

**Fix:** Add 2 tests that mock `subprocess.run` to assert the clamped value reaches the Rust binary:

```python
from unittest.mock import patch, MagicMock
from open_edit.agent.sandbox_bridge import (
    run_free_form, MAX_FREEFORM_TIMEOUT_SEC, MAX_FREEFORM_MEM_MB
)

@patch("open_edit.agent.sandbox_bridge.subprocess.run")
def test_timeout_clamped_to_max(mock_run):
    mock_run.return_value = MagicMock(
        returncode=0,
        stdout='{"exit_code":0,"ok":true,"stderr":""}',
    )
    run_free_form(workdir=tmp_path, code="pass", timeout=999_999)
    cmd = mock_run.call_args[0][0]  # the command list passed to subprocess.run
    assert "--timeout" in cmd
    idx = cmd.index("--timeout")
    assert cmd[idx + 1] == str(MAX_FREEFORM_TIMEOUT_SEC)

@patch("open_edit.agent.sandbox_bridge.subprocess.run")
def test_mem_mb_clamped_to_max(mock_run):
    mock_run.return_value = MagicMock(
        returncode=0,
        stdout='{"exit_code":0,"ok":true,"stderr":""}',
    )
    run_free_form(workdir=tmp_path, code="pass", mem_mb=999_999)
    cmd = mock_run.call_args[0][0]
    assert "--mem" in cmd
    idx = cmd.index("--mem")
    assert cmd[idx + 1] == str(MAX_FREEFORM_MEM_MB)
```

**Why mock `subprocess.run`:** The clamp behavior is at the wrapper level (before the Rust binary is invoked). Mocking `subprocess.run` lets the tests run without bwrap. The test asserts what value is **passed to** the Rust binary — if the clamp is broken, the value sent to the binary exceeds the cap, and the test fails.

**Acceptance:** The clamp behavior is directly tested; a future refactor that breaks the clamp will be caught.

### T7 — IR bootstrap safety net (resolves FR C1 follow-up)

**File:** `open_edit/open_edit/agent/sandbox_bridge.py:339-345` (`_render_bootstrap`), `open_edit/tests/test_sandbox_bridge.py` (existing C1 test)

The C1 fix (already in `5ceb596`) hardcoded the expected list of 24 op names. If a future op is added to `OperationUnion` in `ir/types.py` without updating the `op_types` list in `sandbox_bridge.py`, the regression test only catches it if the test author remembers to update the `expected` list.

**Fix (Pydantic-version-stable):**

`OperationUnion` is `Annotated[Union[...], Field(...)]`. `pydantic.TypeAdapter(OperationUnion).json_schema()` output structure varies across Pydantic v2 minor versions (`oneOf` vs `anyOf`, `$ref` inlining). Use `typing.get_args` instead — pure `typing` introspection, stable forever:

```python
from typing import get_args
from open_edit.ir.types import OperationUnion

# Unwrap Annotated: get_args returns (UnionType, FieldInfo)
_op_union, _ = get_args(OperationUnion)
op_classes = get_args(_op_union)  # 24 classes (or whatever the current count is)
expected_names = {cls.__name__ for cls in op_classes}

# Get the bootstrap source as a string
bootstrap_src = _render_bootstrap(project_id="test", parent_op_id="test")

# Check: every member has a class definition in the bootstrap
for op_name in expected_names:
    assert f"class {op_name}" in bootstrap_src, (
        f"Missing op class definition in bootstrap: {op_name}. "
        f"Either add it to `op_types` in sandbox_bridge.py:_render_bootstrap, "
        f"or this is a regression of the C1 fix."
    )
```

**Why substring match (`class {op_name}`) instead of AST parsing:** AST parsing is more rigorous but the bootstrap is generated from `inspect.getsource(IR)` and contains all 24 op classes' source code (each starts with `class {OpName}(BaseModel):` or similar). A simple substring check is sufficient and fast. If a future op uses a non-class structure (e.g., a Pydantic `create_model` factory), the substring check fails and the test author updates it.

**Acceptance:** Adding a 25th op class to `OperationUnion` causes the test to fail with a clear "Missing op class definition in bootstrap: {new_op}" message. The C1 regression is fully self-enforcing, Pydantic-version-stable, and doesn't depend on a hardcoded count.

## 9. Decision log

| Decision | Why |
|---|---|
| One plan, 9 focused tasks (T0–T8) | The user picked "All review-flagged polish" + "One plan, 7 focused tasks" in brainstorming. T0 (baseline) and T8 (release housekeeping) were added per the FR review; total is still 9 focused tasks with clear scope. |
| T3 excludes `e2e_python_version_mismatch` if env lacks both Pythons | That test needs two Python interpreters on the host; the dev env may not have both. Test-the-env guard. |
| T6 picks (a) for M-6 (remove `--ops-output` flag) instead of (b) wire it up | The flag is dead from the Rust side; the host path is reconstructed from `--scratch`. The 3 callers (Rust CLI parser, Python wrapper at `sandbox_bridge.py:192`, e2e test at `test_free_form_e2e.py:34`) are all updated together. **Verified via T0 baseline grep.** |
| T7 uses `typing.get_args(OperationUnion)` instead of `pydantic.TypeAdapter.json_schema()` | Pydantic v2 schema output varies across minor versions; `get_args` is pure `typing` introspection, stable forever. One extra `get_args` call needed to unwrap the `Annotated` wrapper. |
| T7 uses substring match (`f"class {op_name}" in bootstrap_src`) instead of AST parsing | The bootstrap is generated from `inspect.getsource(IR)` and contains all op class definitions as `class {OpName}(...)`. Substring is sufficient and fast. If a future op uses `create_model`, the substring check fails and the test author updates it. |
| T1 doesn't try to make the 5 e2e tests run on the CI without bwrap | The e2e tests are inherently bwrap-dependent; unskipping them on CI is a separate (out-of-scope) work item. Tracked in v1.2 follow-up. |
| T4 uses `child.wait_with_output()` (not manual pipe reads) | Manual `read_to_string(child.stdout)` then `wait()` deadlocks if the script writes >64KB. `wait_with_output()` reads both pipes concurrently. |
| T2 writes `+memory +cpu` to the **parent's** `cgroup.subtree_control`, not the child's | cgroup v2 semantics: parent's `subtree_control` enables controllers for children. Writing to the child's own file enables controllers for the child's children, not for itself. This is a real bug in the v1.0 spec. |
| T2 uses `chown` for privilege (not systemd-run delegate, not ACLs) | Simplest approach. Resets on reboot — script is designed to be re-run after each boot, or installed as a systemd unit. |
| Parallel execution (3 tracks) over sequential | T2, T5, T7 are fully independent. T6 depends on T4 (both touch main.rs). Track structure cuts wall-clock from ~12h to ~7-8h. |
| Abort criterion: 2× estimated effort per task | Without this, a stuck task can block the release indefinitely. T1 is the most likely to spiral (touches 16+ tests); abort at 4h. |
