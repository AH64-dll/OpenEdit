# Open Edit v1.6 — HyperFrames HTML Overlay Integration (Design)

**Status:** Approved design (brainstorming complete; ready for writing-plans).
**Date:** 2026-07-21
**Prior state:** v1.5 merged to `main` at `ebd106e` (12 commits). 566 passed, 5 skipped, 1 pre-existing deselected. WIP commit `ff51b12` from the parallel agent landed 138 new tests, the HTML overlay IR (`AddHtmlOverlayOp`, `RemoveHtmlOverlayOp`, `HtmlOverlay`), and the canvas timeline panel. v1.5's visual verification loop is in place; the server runs at `0.0.0.0:8000` (PID 640436) with the `pi` LLM provider.

**Outcome target:** when the LLM calls `trigger_render({mode: "overlay"})` and the project's `Timeline` has any HTML overlays, the server composes the MLT background video with an HTML overlay layer rendered by `@hyperframes/engine` (via the `hyperframes` CLI), producing a single MP4 the v1.5 visual verification loop samples and judges. The composited path is **additive** — existing renders (no overlays, or `mode != "overlay"`) are byte-identical to v1.5 behavior.

**Why now:** the v1.5 WIP added HTML overlay IR + a canvas timeline panel + 138 tests, but stopped at the IR layer. The actual rendering pipeline (Chromium → frames → ffmpeg → composite with MLT background) is the missing piece. `@hyperframes/engine` (HeyGen, Apache 2.0, 36.5k⭐, 304 releases, used in production) does exactly this rendering — see `https://github.com/heygen-com/hyperframes`. v1.6 is "don't build the renderer, call the existing one."

---

## 1. Problem

The chat-UI's LLM, given the `AddHtmlOverlayOp` tool, can describe a graphic in HTML/CSS/JS but currently has no way to render it onto the video. Without a render path, overlays stay as data in the edit graph and never become pixels. The v1.5 WIP committed the IR (good groundwork) but no `AddHtmlOverlayOp` ever appears in a final render.

The other AI agent's WIP plan described building a headless-Chromium overlay compositor from scratch (the "HyperFrames-style" approach). That's a 1-2 week build with significant risk. The HeyGen `@hyperframes/engine` is the production-tested implementation of that exact approach — we should call it, not reimplement it.

The LLM needs an end-to-end path: add an `AddHtmlOverlayOp` → call `trigger_render` → the overlay appears in the final MP4 → v1.5's visual verification loop judges whether the LLM's intent matches what was actually rendered.

---

## 2. Architecture overview

The composited render is a 4-stage pipeline that runs concurrently where possible:

```
trigger_render({mode: "overlay"})
   │
   │  pi_bridge._run_trigger_render checks: overlays?  mode=="overlay"?
   │  → if both: composited path. Otherwise: existing MLT path (v1.5).
   │
   ├─ t=0 ─────────────────────────────┐
   │                                  │
   │  STAGE 1: bg render               │  STAGE 2: composition HTML gen
   │  (existing _run_mlt_only_render)  │  (generate_composition_html)
   │  subprocess.run("open_edit        │  pure function, deterministic
   │  render --mode proxy")            │
   │                                  │
   ├─ t=A: comp HTML ready             │
   │  STAGE 3: overlay render starts   │  (Stage 1 still in flight)
   │  subprocess.run(hyperframes)      │
   │  → overlay.mov (ProRes 4444)      │
   │                                  │
   ├─ t=B: both ready                  │
   │  STAGE 4: ffmpeg composite         │
   │  ffmpeg -i bg.mp4 -i overlay.mov   │
   │  -filter_complex overlay          │
   │  -map 0:a -map [outv]             │
   │  → final.mp4                       │
   │
   └─ v1.5 visual verification runs on final.mp4 (unchanged)
```

**Key invariants:**
- v1.5 visual verification is a black box — sees one MP4, samples frames, parses `VERIFICATION: <X>` line. No change to it.
- `mode: "overlay"` is the only opt-in. `trigger_render({})` and `trigger_render({mode: "proxy"})` keep v1.5 behavior byte-identical.
- The bg render is the same subprocess as v1.5 — no parallel `open_edit render` implementations.
- bg render and overlay render run concurrently (`asyncio.create_task` + `asyncio.to_thread`); composite waits for both.

## 3. The composited render path

**Stage 1 — MLT background render (existing, unchanged):**

```bash
open_edit render --mode proxy
  cwd=<project_workdir>
  → <project>/.open_edit/renders/project_*.mp4
```

Returns the bg path. The existing `_run_mlt_only_render` function in `pi_bridge.py` already does this. We just call it from a `bg_renderer` lambda.

**Stage 2 — Composition HTML generation (pure function):**

`html_overlay.generate_composition_html(timeline, project_workdir, render_spec) -> str` builds the HyperFrames composition HTML. Pure function — no subprocess, no I/O, deterministic. The caller writes the string to a temp file and passes the path to Stage 3. See §4 for the gotchas (transparent body, HTML escape, JSON escape, path validation).

**Stage 3 — HyperFrames overlay render (subprocess):**

```bash
node_modules/.bin/hyperframes render \
  --input composition.html \
  --output overlay.mov \
  --format mov \
  --width <W> --height <H> --fps <FPS>
```

`--format mov` produces ProRes 4444 (the only format that carries alpha through ffmpeg's `overlay` filter reliably — see §5 for why). No `--transparent` flag (that's for WebM/png-sequence, not MOV). Width/height/fps come from the same `RenderSpec` that drives the bg render — no re-derivation.

**Stage 4 — ffmpeg composite (subprocess):**

```bash
ffmpeg -y \
  -i bg.mp4 \
  -i overlay.mov \
  -filter_complex "[0:v][1:v]overlay=eof_action=pass" \
  -map 0:a \
  -map [outv] \
  -c:a copy \
  -c:v libx264 \
  final.mp4
```

`-map 0:a -map [outv]` is explicit so bg's audio wins (ffmpeg's auto-selection can be wrong). `-c:a copy` avoids re-encoding audio. `overlay=eof_action=pass` lets the shorter stream's last frame persist (handles duration drift between bg and overlay layers).

**Stage 5 — Visual verification (v1.5, unchanged):**

The agent loop's `_maybe_verify_render` runs on `final.mp4` exactly as it does on v1.5. The LLM sees sampled frames, must emit `VERIFICATION: PASS|FAIL|UNCERTAIN`. No changes here.

## 4. Composition HTML generation

**The composition contract** — one root `<div data-composition-id="...">`, one `<div class="clip" data-track-index="N">` per overlay, all overlays share `window.__open_edit_vars` and the project templates directory.

**Generated HTML shape (one overlay at t=2.0s, 5s long, 1920x1080 @30fps):**

```html
<!doctype html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <style>
    /* CRITICAL: no background on html, body, or root. */
    html, body, [data-composition-id] { background: transparent; margin: 0; padding: 0; }
    .clip { position: absolute; inset: 0; }
  </style>
</head>
<body>
  <div data-composition-id="open_edit_overlay" data-width="1920" data-height="1080">
    <div class="clip" data-start="2.0" data-duration="5.0" data-track-index="0"
         id="overlay_a1b2c3">
      <!-- contents of templates/lower_third.html, with {{title}} escaped via html.escape -->
      <div class="lower-third">…&lt;script&gt;alert(1)&lt;/script&gt;…</div>
    </div>
  </div>
  <script>
    /* HyperFrames computes total render length from clip children.
       Root data-duration is intentionally absent. */
    window.__open_edit_vars = {"a1b2c3": {"title": "<\/script><script>alert(1)<\/script>"}};
  </script>
</body>
</html>
```

**The five gotchas (security + correctness):**

1. **Transparent body** — explicit `background: transparent` on `html`, `body`, and the root composition div. No `background-color` anywhere. The renderer must produce transparent pixels everywhere except the overlay elements themselves.
2. **No root `data-duration`** — HyperFrames computes total length from clip children. Adding a root `data-duration` is dead weight that contradicts the per-clip model and would cause off-by-one in the frame budget. The root carries only `data-composition-id`, `data-width`, `data-height`.
3. **HTML-escape `{{key}}` substitution** — every primitive value goes through `html.escape(value, quote=True)` before substitution. Non-primitive values (dicts, lists) skip `{{}}` substitution entirely; they only appear in the JS object. This blocks `<script>alert(1)</script>` in a title from executing in headless Chrome.
4. **Script-safe JSON for `window.__open_edit_vars`** — `json.dumps(obj).replace("<", "\\u003c").replace("&", "\\u0026").replace("\u2028", "\\u2028").replace("\u2029", "\\u2029")`. Defends against `</script>` injection, HTML entity confusion, and the JS line-separator parsing bug (U+2028/U+2029 are valid JSON but illegal unescaped in JS string literals pre-ES2019).
5. **Path traversal guard on `template_path`** — reject absolute paths and `..` segments. LLM-supplied `template_path` is untrusted.

**Template resolution order (validated against path traversal):**

```python
def _resolve_template_path(template_path: str, project_workdir: Path) -> Path:
    if Path(template_path).is_absolute():
        raise OverlayRenderError(f"absolute template_path not allowed: {template_path}")
    if ".." in Path(template_path).parts:
        raise OverlayRenderError(f"template_path with '..' not allowed: {template_path}")
    candidate = (project_workdir / template_path).resolve()
    if candidate.is_file():
        return candidate
    builtin = (Path(__file__).parent / "templates" / "overlay" / template_path).resolve()
    if builtin.is_file():
        return builtin
    raise OverlayRenderError(f"template_not_found: {template_path}")
```

**Track assignment:** overlays sorted by `position_sec`; a new `data-track-index` is allocated only when the previous overlay on that track hasn't ended. Non-overlapping overlays share a track; overlapping overlays get distinct tracks. Zero-indexed.

**Variable dual-exposure:** server substitutes `{{key}}` placeholders for primitive values (so the LLM doesn't write JS for static text); `window.__open_edit_vars[overlay_id]` exposes the full dict (including nested data) for the template's JS to read.

**Edge cases (table):**

| Case | Behavior |
|---|---|
| `{{key}}` value contains `<` / `>` / `&` / `"` | Escaped via `html.escape(value, quote=True)` |
| Variable value contains `</script>` | Escaped to `<\/script>` in the JSON blob |
| Variable is a dict/list | Only goes into the JS object; `{{}}` substitution skipped |
| `template_path` is absolute | Rejected — `OverlayRenderError("absolute template_path not allowed: ...")` |
| `template_path` contains `..` | Rejected — `OverlayRenderError("template_path with '..' not allowed: ...")` |
| `template_path` resolves to a symlink outside the dirs | Rejected by `resolve()` + `is_file()` |
| `template_path` not found in either dir | `OverlayRenderError("template_not_found: ...")`; `pi_bridge` falls back to bare MLT |
| Empty `variables` | `window.__open_edit_vars[overlay_id] = {}`; `{{}}` is a no-op |
| Missing `{{key}}` in `variables` | Literal `{{key}}` left in HTML; runtime warning logged |
| Template HTML is malformed | Linter is NOT called from Python; HyperFrames lints at render time |
| Two overlays at the same `position_sec`, same duration | Different `data-track-index`, render on separate tracks |
| Composition total duration | Derived from clip children by HyperFrames; we don't set root `data-duration` |

**Built-in templates shipped with v1.6:** just `templates/overlay/lower_third.html` (name + title bar, two color variables, no JS). The catalog expansion (title cards, captions, data charts, animated GSAP templates) is deferred to v1.6.x.

**Forward-looking (catalog/v1.6.x — not v1.6):** animated templates using GSAP/Lottie need a frame adapter registered on the composition for HyperFrames to seek deterministically:

```html
<script src="https://cdn.jsdelivr.net/npm/gsap@3/dist/gsap.min.js"></script>
<script>
  const tl = gsap.timeline({ paused: true });
  tl.from(".lower-third", { y: 40, opacity: 0, duration: 0.4 }, 0);
  window.__timelines = window.__timelines || {};
  window.__timelines["open_edit_overlay"] = tl;
</script>
```

`window.__timelines[id]` with `{paused: true}` is what the engine seeks against (from HyperFrames' own contract). Lands in `docs/superpowers/specs/v1.6.x-catalog.md` when we add the first animated template.

## 5. HyperFrames subprocess + ffmpeg composite

**The pinned binary with `npx` fallback:**

```python
def _resolve_hyperframes_bin() -> str:
    pinned = Path("node_modules/.bin/hyperframes")
    if pinned.is_file():
        return str(pinned)
    _LOG.warning(
        "hyperframes pinned binary not found at %s; falling back to npx hyperframes "
        "(network resolution + version drift risk). Run npm install at repo root.",
        pinned,
    )
    return "npx hyperframes"
```

First-use WARNING (not silent) when the fallback fires. The env var `OPEN_EDIT_HYPERFRAMES_BIN` overrides the default; if set, that path is used directly with no fallback.

**`render_overlay_layer` — the HyperFrames CLI call:**

```python
cmd = shlex.split(render_spec["hyperframes_bin"]) + [
    "render",
    "--input", str(composition_html_path),
    "--output", str(overlay_path),
    "--format", "mov",       # ProRes 4444 for alpha
    "--width", str(render_spec["width"]),
    "--height", str(render_spec["height"]),
    "--fps", str(render_spec["fps"]),
]
proc = subprocess.run(
    cmd, capture_output=True, text=True,
    timeout=render_spec["hyperframes_timeout_s"],
    shell=False, check=False,
)
```

Three error paths wrapped in `OverlayRenderError`: `FileNotFoundError` (binary missing), `TimeoutExpired` (after `hyperframes_timeout_s`, default 120s), `returncode != 0` (CLI failed; stderr/stdout captured).

**Why `--format mov` and not `--transparent`:** H.264 has no alpha channel — full stop. MP4/H.264 cannot carry transparency. HyperFrames documents transparency via output format: `--format mov` (ProRes 4444) or `--format webm` (VP9 with alpha) or `--format png-sequence`. ProRes 4444 is the explicit recommendation for the compositing workflow because ffmpeg's ProRes decoder surfaces alpha natively; WebM/VP9 alpha is fragile (most non-browser tools render transparent regions as black). PNG-sequence is a frame-storm — wrong for a single composited MP4.

**The PRO_RES tradeoff (worth knowing):** ProRes 4444 MOVs run noticeably heavier than H.264 (tens of MB for short clips vs a few MB). Budget for temp-disk IO if a session renders many overlays. The intermediate `overlay.mov` lives in `tmpdir` and is cleaned up after the composite.

**`composite_with_background` — the ffmpeg call:**

```python
cmd = [
    "ffmpeg", "-y",
    "-i", str(bg_path),       # [0:v] = bg video, [0:a] = bg audio
    "-i", str(overlay_path),  # [1:v] = overlay (ProRes 4444 with alpha)
    "-filter_complex", "[0:v][1:v]overlay=eof_action=pass",
    "-map", "0:a",            # bg's audio wins — explicit, not auto-selected
    "-map", "[outv]",         # the composited video
    "-c:a", "copy",           # don't re-encode audio
    "-c:v", "libx264",        # final encode
    str(output_path),
]
```

`eof_action=pass` lets the shorter stream's last frame persist — critical for the case where `overlay.mov` ends before `bg.mp4` (a lower third that disappears cleanly instead of freezing on screen).

**W/H and duration source of truth:** the `RenderSpec` is built once in `pi_bridge` from the MLT project profile. The same spec is passed to Stage 1 (bg render), Stage 3 (hyperframes `--width/--height/--fps`), and Stage 4 (ffmpeg — implicit, because the `-i` files have the right dimensions). No re-derivation anywhere.

## 6. Concurrent orchestration

**`render_composited` — the async orchestrator:**

```python
async def render_composited(
    timeline, project_workdir, render_spec, bg_renderer, should_cancel=None
) -> Path:
    """Concurrent pipeline. Raises OverlayRenderError on failure; the
    exception carries bg_path when the bg render succeeded (so pi_bridge's
    fallback reuses the completed bg without re-running the MLT encode)."""
    bg_task = asyncio.create_task(asyncio.to_thread(bg_renderer))
    if should_cancel and should_cancel():
        bg_task.cancel(); raise _cancelled("bg render", None)
    comp_html = await asyncio.to_thread(generate_composition_html, timeline, project_workdir, render_spec)
    comp_path = render_spec["tmpdir"] / "open_edit_composition.html"
    await asyncio.to_thread(comp_path.write_text, comp_html, encoding="utf-8")
    try:
        overlay_path = render_spec["tmpdir"] / "overlay.mov"
        overlay_task = asyncio.create_task(
            asyncio.to_thread(render_overlay_layer, comp_path, overlay_path, render_spec, should_cancel)
        )
        bg_path = await bg_task
        if should_cancel and should_cancel():
            overlay_task.cancel(); raise _cancelled("overlay render", bg_path)
        overlay_result_path = await overlay_task
        if should_cancel and should_cancel():
            raise _cancelled("ffmpeg composite", bg_path)
        final_path = render_spec["tmpdir"] / "composited.mp4"
        await asyncio.to_thread(
            composite_with_background, bg_path, overlay_result_path, final_path, render_spec, should_cancel
        )
        return final_path
    finally:
        comp_path.unlink(missing_ok=True)
```

**Concurrency analysis:** the bg render (Stage 1) runs in parallel with the composition HTML generation (Stage 2) and the overlay render (Stage 3). The final ffmpeg composite (Stage 4) waits for both. Worst case (bg is the slowest stage): total = bg_time + composite_time. Best case (bg is fast): total = max(bg, comp_html + overlay) + composite. The savings is the comp_html + overlay duration — meaningful on any non-trivial project (typical bg is 30s+, comp_html + overlay is 5-10s).

**`pi_bridge._run_trigger_render` — the whole diff:**

```python
if _should_use_composited(args, project_path, render_spec):
    try:
        return await html_overlay.render_composited(
            timeline, project_path, render_spec,
            bg_renderer=lambda: _run_mlt_only_render(args, project_path, render_spec),
            should_cancel=should_cancel,  # plumbed from v1.5 agent loop
        )
    except html_overlay.OverlayRenderError as e:
        log.warning(
            "overlay render failed, returning %s: %s",
            "MLT bg" if e.bg_path else "fallback MLT",
            e,
        )
        return e.bg_path or _run_mlt_only_render(args, project_path, render_spec)
```

~15 lines. The `bg_renderer` lambda is the only "extra" orchestration in `pi_bridge`; everything else lives in `html_overlay`. The `bg_path` on the exception prevents the duplicate bg render on the common failure path (HyperFrames missing or failed).

**`OverlayRenderError` carries the bg path on failure:**

```python
class OverlayRenderError(Exception):
    """Raised when the composited render fails. Carries bg_path when the
    bg render succeeded but a downstream step failed — lets pi_bridge's
    fallback reuse the completed bg without re-running the MLT encode.
    """
    def __init__(self, message: str, bg_path: Path | None = None):
        super().__init__(message)
        self.bg_path = bg_path
```

## 7. Cancellation (3 stages, all plumbed)

The v1.5 agent loop's WS-disconnect handler exposes a `should_cancel: Callable[[], bool]`. v1.6 plumbs this through to all three subprocess calls. Each subprocess wrapper polls `should_cancel` before and after the call; when it returns True, the child process is killed.

| Stage | When cancel fires | What happens | `bg_path` on exception |
|---|---|---|---|
| bg render | mid-`_run_mlt_only_render` | bg subprocess killed; `OverlayRenderError(bg_path=None)` | None — bg wasn't done |
| HyperFrames overlay | mid-`render_overlay_layer` | hyperframes killed; `OverlayRenderError(bg_path=<bg>)` | bg is done — pi_bridge returns it without re-encoding |
| ffmpeg composite | mid-`composite_with_background` | ffmpeg killed; `OverlayRenderError(bg_path=<bg>)` | bg + overlay done; pi_bridge returns bg (no overlay) |

The two later cases are the win: in v1.5, only bg cancellation existed, and a mid-overlay cancel had no defined behavior. v1.6 covers all three with consistent `bg_path` semantics.

## 8. Components (the public surface of `html_overlay.py`)

**Four public functions + one exception:**

```python
class OverlayRenderError(Exception):
    def __init__(self, message: str, bg_path: Path | None = None): ...

class RenderSpec(TypedDict):
    width: int
    height: int
    fps: int
    duration_sec: float
    mode: Literal["proxy", "final"]
    hyperframes_bin: str
    hyperframes_timeout_s: int
    tmpdir: Path

def generate_composition_html(
    timeline: Timeline, project_workdir: Path, render_spec: RenderSpec
) -> str: ...

def render_overlay_layer(
    composition_html_path: Path, output_path: Path, render_spec: RenderSpec,
    should_cancel: Callable[[], bool] | None = None
) -> Path: ...   # raises OverlayRenderError

def composite_with_background(
    bg_path: Path, overlay_path: Path, output_path: Path, render_spec: RenderSpec,
    should_cancel: Callable[[], bool] | None = None
) -> Path: ...   # raises OverlayRenderError

async def render_composited(
    timeline: Timeline, project_workdir: Path, render_spec: RenderSpec,
    bg_renderer: Callable[..., Path], should_cancel: Callable[[], bool] | None = None
) -> Path: ...
```

**`RenderSpec` provenance:** `get_mlt_profile(project_path) -> RenderSpec` lives in `open_edit/render/emitter.py` (the same module that builds the MLT XML for Stage 1). `pi_bridge` calls it once, passes the result to Stage 1, Stage 3, and Stage 4. No new parsing logic in `pi_bridge`.

**`tmpdir` lifetime:** `tempfile.mkdtemp(prefix="open_edit_overlay_", dir=<tmpdir_base>)` at the start of `render_composited`; `shutil.rmtree(tmpdir)` in the outermost `finally` (cleaned up by GC even on uncaught exceptions). Each render gets its own subdir; no cross-render pollution.

**`bg_renderer` signature:** takes no arguments, returns a `Path` (the bg MP4). `pi_bridge` passes a `lambda: _run_mlt_only_render(args, project_path, render_spec)`. `render_composited` calls it once via `asyncio.to_thread(bg_renderer)`.

## 11. Acceptance criteria (13 end-to-end)

1. **Basic compositing** — add `AddHtmlOverlayOp(template_path="overlay/lower_third.html", variables={"title": "Hello"}, position_sec=2.0, duration_sec=5.0)` to a project with at least one video clip; call `trigger_render({mode: "overlay"})`; the resulting MP4 shows the lower third over the bg video for the 2-7s window.
2. **HTML escape** — add an overlay with `variables={"title": "<script>alert(1)</script>"}`; the composited MP4 shows the title as literal text, no script execution, no broken render.
3. **JSON escape (the `</script>` break)** — add an overlay with `variables={"title": "before</script><script>alert(2)</script>after"}`; the composited MP4 shows the title as `before</script>...after` (escaped to `<\/script>`), no script execution, no broken render.
4. **Multi-overlay track assignment** — two overlapping overlays at the same time position; both visible in the final MP4, on different tracks.
5. **HyperFrames missing fallback** — remove `node_modules/.bin/hyperframes`; call `trigger_render({mode: "overlay"})`; the request succeeds (returns the bare MLT MP4 as fallback), a WARNING is logged: `hyperframes pinned binary not found at <path>; falling back to npx hyperframes (network resolution + version drift risk). Run npm install at repo root.`
6. **Mode is opt-in** — call `trigger_render({})` (no `mode: "overlay"`) on a project with overlays; the bare MLT path runs (no overlay, no error).
7. **v1.5 verification still works** — after a composited render, the v1.5 visual verification loop runs unchanged; the LLM is shown sampled frames of the composited MP4 and must emit `VERIFICATION: PASS|FAIL|UNCERTAIN`.
8. **bg render cancellation** — `WebSocketDisconnect` mid-bg → bg subprocess killed, `OverlayRenderError(bg_path=None)` raised, pi_bridge re-runs the bare MLT path.
9. **overlay render cancellation** — `WebSocketDisconnect` mid-HyperFrames → hyperframes killed, `OverlayRenderError(bg_path=<bg>)` raised, pi_bridge returns the bg path (no duplicate encode).
10. **ffmpeg composite cancellation** — `WebSocketDisconnect` mid-composite → ffmpeg killed, `OverlayRenderError(bg_path=<bg>)` raised, pi_bridge returns the bg path.
11. **Pinned binary wins** — after `npm install`, the first render logs nothing about a fallback; `_resolve_hyperframes_bin()` returns the pinned path. `npx hyperframes` is never invoked.
12. **Path traversal rejected** — `AddHtmlOverlayOp(template_path="../../../etc/passwd", ...)` → `OverlayRenderError("template_path with '..' not allowed: ../../../etc/passwd")`; no fallback (the op itself fails).
13. **No regression in v1.5 tests** — the 13 visual-verify + 5 chip + 4 llm-usage + 7 pi-bridge tests still pass without modification.

## 12. Out of scope (deferred to v1.6.x+)

- **Built-in template catalog beyond `lower_third.html`.** Title cards, captions, data charts, animated GSAP/Lottie templates. Lands in `docs/superpowers/specs/v1.6.x-catalog.md`.
- **Persistent Node sidecar.** The 2-3s per-render Chrome launch is acceptable for v1.6. Revisit if render latency becomes a bottleneck (concurrent renders to multiple users, CI, etc.).
- **Audio mixing via HyperFrames.** We keep our existing audio path (carried via `-c:a copy` from the bg MP4). HyperFrames has audio mixing via `<audio>` element parsing but we don't need it.
- **AWS Lambda rendering.** v1.6 is local-only. Distributed rendering is v1.6.x+ if/when there's a use case.
- **Full pipeline replacement.** v1.6 is overlay-only. Replacing our entire MLT render with HyperFrames is a v2+ conversation, not a v1.6.x decision.
- **Remotion-to-HyperFrames porting.** We have an IR, not Remotion. Out of scope.

**Strategic note (for the user):** v1.6 deliberately does NOT make MLT removable. The "Overlay-only (surgical)" scope locks us into MLT + HyperFrames coexistence. If we ever want to drop MLT entirely (e.g., to support 4K+, non-ML-native formats, or a fundamentally different compositor), that's a v2+ decision, not a v1.6.x decision.
