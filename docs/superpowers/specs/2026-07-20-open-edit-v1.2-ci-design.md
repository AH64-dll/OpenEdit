**Status:** Draft (brainstorming complete; pending user review)
**Date:** 2026-07-20
**Supersedes:** n/a

**Outcome target:** GitHub Actions CI that un-ignores the 5 Rust integration tests and 5 Python e2e tests added in v1.1, and runs all 308+ Python + 1+ Rust + 6 Rust integration + 5 e2e tests on every push to main and every PR.

## 1. Goal

Bring v1.2 to life: set up CI infrastructure that un-ignores the bwrap-dependent tests written in v1.1 T3, so the v1.2 follow-up filed at `.superpowers/sdd/v1.2-followup.md` is no longer deferred.

**Why GitHub Actions (and not the user's self-hosted option):**
- The user picked GitHub Actions in brainstorming. It's the most portable format (Forgejo/Gitea can run the same YAML with minor tweaks).
- The project currently has no remote configured. GitHub Actions config sits in the repo, ready to use when the user pushes to GitHub.
- Free for public repos; reasonable minutes for private.

## 2. Architecture

**One workflow file:** `.github/workflows/ci.yml`

**Three jobs, all on `ubuntu-latest` (bwrap + cgroup v2 + passwordless sudo):**

1. **`python-unit`** (~2 min) — runs the 308 Python unit tests. No special infra. Matrix on Python 3.11, 3.12, 3.14.
2. **`rust-build`** (~1 min) — `cargo build` + `cargo test` for the Rust sandbox. No special infra. Linux only.
3. **`bwrap-tests`** (~5-10 min) — un-ignores the bwrap-dependent tests. Runs:
   - `cargo test --features integration` (1 pass + 6 ignored, with the 5 new un-ignored)
   - The 5 Python e2e tests (currently skipped, un-skipped here)
   - Uses the `bin/setup_render_cgroup.sh` script with `sudo` to set up cgroup v2.
   - Requires `bwrap` and `sudo` (both available on `ubuntu-latest`).

**Triggers:**
- `push` to `main`
- `pull_request` to `main`
- `workflow_dispatch` (manual trigger from GitHub UI)
- `tags` of the form `v*` (e.g., `v1.1.0`) — for release verification

**Concurrency:**
- Cancel in-progress runs when a new commit is pushed to the same PR.
- Don't cancel runs on `main` (they're authoritative).

**Permissions:**
- `contents: read` (default; just clone the repo).
- No special tokens or secrets needed.

## 3. Components

### Job 1: `python-unit`

**Matrix:** `python-version: [3.11, 3.12, 3.14]`

**Steps:**
1. Checkout
2. Set up Python (`actions/setup-python@v5`)
3. Cache pip (`actions/cache@v4`, key: `pip-${{ runner.os }}-${{ matrix.python-version }}-${{ hashFiles('**/pyproject.toml') }}`)
4. Install dependencies: `pip install -e "open_edit[dev]"`
5. Run pytest: `cd open_edit && pytest tests/ --ignore=tests/test_free_form_e2e.py -v`
   - `--ignore=tests/test_free_form_e2e.py` excludes the 5 bwrap-dependent e2e tests from this job (they run in `bwrap-tests`).

**Expected:** 308 passed, 0 skipped (per Python version).

### Job 2: `rust-build`

**Single version, Linux only.**

**Steps:**
1. Checkout
2. Set up Rust toolchain (`dtolnay/rust-toolchain@stable`)
3. Cache cargo (`Swatinem/rust-cache@v2`)
4. Build: `cd open_edit/sandbox && cargo build --release`
5. Test: `cargo test` (1 passed, 1 ignored — `render_sandbox_runs_python_writes_output` requires bwrap + cgroup, runs in `bwrap-tests`)

**Expected:** 1 passed, 1 ignored (Linux).

### Job 3: `bwrap-tests`

**Single version, Linux only.** Runs after `python-unit` and `rust-build` pass (to save CI minutes).

**Steps:**
1. Checkout
2. Set up Python 3.14 (the pinned version per `sandbox_bridge.py`)
3. Set up Rust toolchain
4. Cache pip + cargo
5. Install system deps: `sudo apt-get update && sudo apt-get install -y bubblewrap`
   - bwrap is usually pre-installed on ubuntu-latest, but this is defensive.
6. Run the cgroup setup script: `sudo bin/setup_render_cgroup.sh`
   - Idempotent; safe to re-run.
7. Install Python deps: `pip install -e "open_edit[dev]"`
8. Build Rust: `cd open_edit/sandbox && cargo build --release`
9. **Run un-ignored Rust integration tests:** `cargo test --features integration -- --include-ignored`

## 4. Data flow

**No persistent data between jobs.** Each job is a fresh `ubuntu-latest` runner.

**Cross-job dependencies:**
- `bwrap-tests` depends on `python-unit` and `rust-build` succeeding (`needs: [python-unit, rust-build]`).
- This prevents running the slow bwrap tests if a fast lane fails.

## 5. Error handling

**What happens when a job fails:**
- The whole workflow run fails.
- GitHub displays the failing log.
- A PR cannot be merged if a required check fails (this is the "merge gate" benefit).

**What happens when the cgroup setup script fails (T2's edge case: "no internal processes" in root cgroup):**
- The script exits 1 with a clear error message.
- The `bwrap-tests` job fails.
- The failure shows up in the CI log with the clear error.
- We accept this — if cgroup v2 is misconfigured in CI, we want to know, not silently pass.

**What happens when bwrap is missing (defensive install):**
- `sudo apt-get install -y bubblewrap` installs it.
- If installation fails, the job fails with the apt error.
- This should be rare on `ubuntu-latest` (bwrap is usually pre-installed).

## 6. Testing

### Local validation
Before pushing, the user can:
1. Install `act` (https://github.com/nektos/act) to run GitHub Actions locally.
2. Run `act -j python-unit` to verify the Python job.
3. Run `act -j rust-build` to verify the Rust job.
4. Run `act -j bwrap-tests` (requires Docker with `--privileged` for bwrap + cgroup).

### First-push validation
When the user pushes to GitHub:
1. The workflow runs on the push.
2. The user can watch the run in the Actions tab.
3. If the `bwrap-tests` job fails on `ubuntu-latest` (which should be rare), the failure is visible in the log.

### Ongoing validation
- The 5 Rust tests + 5 Python e2e tests will run on every PR.
- The whole workflow takes ~7-12 min per run (parallel jobs cut the wall-clock).

## 7. Out of scope

- **Matrix on multiple OS** (macOS, Windows). v1 is Linux-only per the spec.
- **Code coverage reporting** (codecov, coveralls). v1.2 is just the "run the tests" infra.
- **Linting** (ruff, clippy). Already in plan scope; can be added as separate jobs in a v1.3 follow-up.
- **Release workflow** (auto-publish to PyPI, build binaries). v1.3 follow-up.
- **Self-hosted runner setup.** If the user later moves to Forgejo/Gitea, they'll need to write a parallel config or migrate this one.

## 9. Decision log

| Decision | Why |
|---|---|
| GitHub Actions (not self-hosted) | User picked it. Most portable; works on GitHub, Forgejo, Gitea. |
| 3 jobs (python-unit, rust-build, bwrap-tests) | Separates fast unit tests from slow bwrap tests. bwrap-tests depends on the others succeeding — saves CI minutes. |
| `bwrap-tests` runs on `ubuntu-latest` (not a self-hosted runner) | ubuntu-latest has bwrap + cgroup v2 + passwordless sudo. Simpler than maintaining a custom runner. |
| Python matrix 3.11, 3.12, 3.14 | Project requires >=3.11. The pinned version is 3.14. 3.12 is the current Debian/Ubuntu default; testing it catches "only works on my Python" bugs. |
| `--include-ignored` for Rust | Standard cargo flag. The 5 new tests from v1.1 T3 are `#[ignore]`-d; this flag un-ignores them. |
| `pytest tests/test_free_form_e2e.py` directly in bwrap-tests | The 5 e2e tests are skipped by a `@pytest.mark.skip` that checks for bwrap availability. In a bwrap-capable env, they run. The skip happens in `test_free_form_e2e.py:146`. |
| `--ignore=tests/test_free_form_e2e.py` in python-unit | The 5 e2e tests need bwrap, which is unavailable in the fast python-unit job. Explicitly exclude. |
| Cancel in-progress runs on PRs (concurrency) | Saves CI minutes when developers push rapidly. Don't cancel on main. |

## 10. Risks

| Risk | Mitigation |
|---|---|
| bwrap-tests fails on ubuntu-latest due to env issues | The bwrap-tests job logs are clear; failures show the exact command. The `bin/setup_render_cgroup.sh` script has clear error messages for the cgroup "no internal processes" edge case. |
| `act` local testing differs from real GitHub Actions | We document the difference; `act` is for local development, real CI is authoritative. |
| Test counts drift (new tests added) | The pytest / cargo test commands don't hardcode counts; they just assert success. Adding tests doesn't break CI. |
| The 5 e2e tests have been skipped for a long time and may have bugs | This is exactly why un-skipping them in CI is valuable. If they fail, the failure is visible. |
| `bwrap` permission errors in CI | The 5 Rust tests need `--share-net` or similar bwrap args. If they fail with permission errors, the failure is visible; we can iterate on the bwrap invocation. |
| `melt` (MLT framework) not available in ubuntu-latest | The 5 e2e tests use `melt`. We add `sudo apt-get install -y melt` to the bwrap-tests job. |
| Cost (CI minutes) | bwrap-tests is the only slow job. ~5-10 min × number of PRs. Free for public repos. For private, this is ~$0.08/min × ~7-12 min × N PRs. |
