# v1.2.1 followup (post-release)

**Status:** DISCOVERED in v1.2.0 use, FIXED on main in commit 8d058fc.

## Bug

`open_edit free-form` fails with:
  internal_error: AttributeError("'NoneType' object has no attribute 'get'")

## Root Cause

`open_edit/open_edit/agent/sandbox_bridge.py` `_run_sandboxed` does not
pass `--ops-output <ops_path>` to the Rust sandbox binary. The Rust
binary's CLI still requires this flag (no default). The binary exits
with a usage error to stderr, the bridge gets no JSON on stdout, and
crashes at `rust.get('ok')` because `rust` is `None`.

The flag was removed in v1.1 T6 cleanup (commit 0567448) — flagged as
"unused" because no test exercised the full subprocess call path.
The 5 free-form tests in `test_free_form_e2e.py` skip when bwrap is
missing, so this never failed locally.

## Impact

v1.2.0 release is broken for any user who:
- Has the Rust binary installed AND
- Calls `open_edit free-form`

In practice: every user with a working build (since the binary is
required for free-form). The bwrap tests would catch this in CI, but
the test suite was the very thing that was broken.

## Fix (already on main, commit 8d058fc)

1. Pass `--ops-output <ops_path>` in the subprocess call.
2. Defense in depth: when `rust` is None, return `sandbox_protocol_error`
   with a clear message (no more AttributeError).

## Tests

- `test_run_free_form_no_json_in_stdout_returns_protocol_error`: defense

## Why wasn't this caught earlier

- v1.1 T6 cleanup removed the flag with a test that asserted the
  subprocess.cmd was missing the flag (the brief literally asked to
  remove it as "unused"). The brief was wrong; the test was
  implementation-biased, not behavior-validating.
- v1.1 final review did not exercise the free-form path end-to-end.
- v1.2 final review did not exercise the free-form path end-to-end.
- The 5 free-form tests skip in this dev env (bwrap missing).
- The smoke test (in this session) used the programmatic API, not
  `open_edit free-form`.

## Lesson

A "cleanup" PR that removes a CLI flag from a subprocess call is a
behavior change, not just an import cleanup. TDD would have caught
this if the brief asked "behavior: free-form runs successfully with
ops.jsonl output" instead of "remove unused import".
