# v1.3 followup — chat UI backend integration + Pi LLM provider

**Status:** DONE. Includes both the chat-UI backend integration and the
Pi LLM provider (so the chat can talk to a real LLM).

## Part 1: Chat-UI backend integration

Integrated the chat-UI backend (`open_edit/serve/`) that the user's external
model produced. The user's model generated ~2800 lines of FastAPI + WebSocket
+ agent-loop + frontend code, but it had three critical mismatches with the
real Open Edit codebase. We fixed all three.

## Mismatches found and fixed

### 1. Schema mismatch (CRITICAL)

The generated `serve/projects.py` queried tables that don't exist in the
real Open Edit schema:

- Generated: `assets (hash, filename, duration)`, `ops (id, type, payload, ...)`,
  `effects`, `review_notes (...)`
- Real (`open_edit/storage/schema.sql`): `edits (edit_id, parent_id, kind, author, timestamp, status, sequence_num, payload)`, `project_meta`, `jobs`
- Real notes (`open_edit/storage/notes.py:80-100`): `notes (note_id, project_id, anchor_type, anchor, text, source, status, ...)`
- Real assets: CAS filesystem layout at `.open_edit/assets/<prefix>/<hash>` with sidecar `<hash>.meta.json` (no SQL table)

**Fix:** Rewrote `serve/projects.py` to use the real storage classes:
`AssetStore` (filesystem), `EditGraphStore.load_all()` (ops), `NotesStore.list_all()`
(notes). Dropped the invented SQL queries and the `_bootstrap_minimal_db`
fallback that created wrong tables.

### 2. Invented tools (HIGH)

The generated `serve/tool_schemas.py` listed 12 real tools + 1 virtual.
Only 10 real tools exist (`open_edit/agent/tools/pyagent_*.py`):

- ✅ 10 real: `add_marker, analyze_narrative, generate_visual_for_segment, get_pending_notes, get_style_profile, place_sfx, propose_silence_cuts, run_python, select_music, set_pinned_value`
- ❌ 2 invented: `get_asset_info, list_assets` — `agent.py:169-171` would have raised `RuntimeError: tool not found` when the LLM called them.

**Fix:** Removed the 2 invented tools. Now 11 tools (10 real + 1 virtual
`trigger_render`).

### 3. IR model mismatch (MEDIUM)

The generated `serve/tool_schemas.py:_IR_MODEL_SUMMARY` described a 4-op
model (AddClipOp, AddEffectOp, RemoveOp, UpdateOp). Real Open Edit has
**24 op kinds** in `open_edit/ir/types.py`.

**Fix:** Rewrote the IR summary to enumerate all 24 real op kinds, their
payload shapes, and the common fields every op inherits from `Operation`.

## Part 2: Pi LLM provider (subprocess-based, no API key)

After the chat UI was working with mocked LLMs, the user wanted to
attach the **pi** CLI (Mario Zechner's pi-coding-agent) as the LLM
provider. Pi is a local CLI from `npm` that talks to multiple model
providers (Anthropic, OpenAI, OpenCode, etc.) and supports a
TypeScript extension system for tools.

### Architecture

```
open_edit.serve.agent.run_agent_turn()
    ↓
open_edit.serve.llm.stream_chat(provider="pi")
    ↓ spawns subprocess
pi --provider opencode-go --model X --mode json \
   --extension open_edit/serve/pi_extension/extension.ts \
   --session-id <conv_id> \
   --print <user_message> \
   --append-system-prompt <system>
    ↓
pi_extension/extension.ts (registers 11 tools)
    ↓ subprocess per tool call
python -m open_edit.serve.pi_bridge --tool <name> \
   --project <path> --args '{...}'
    ↓
open_edit.agent.tools.<name>(args, project_path)
    ↓
real edit_graph.db / notes.db
```

### Why the TS extension approach

Pi's CLI only allows **enabling/disabling** tools by name; it has no
flag to register new ones. Adding new tools requires a TypeScript
extension. This matches the user's existing pattern in
`pyagent-kdenlive-guide/phase3_pyagent_core/extension.ts`.

### Key design decisions

- **Tool execution happens in the extension, not in our agent loop.**
  When the LLM calls a tool, pi's extension runs it via the bridge
  subprocess and returns the result. Our agent loop just consumes the
  `tool_result` event and forwards it to the client. The agent loop
  detects the `pi` provider via `llm._provider()` and skips its own
  tool execution path.
- **Conversation history is in pi's session**, not in our
  `conversation_history` list. We pass the same `conv_id` as pi's
  `--session-id` so all turns land in the same session.
- **System prompt is appended on every turn** via
  `--append-system-prompt`. This updates the LLM's view of project
  state without losing history. Trade-off: prompt grows over time.
  Future: use `@file` references to keep the prompt slim.
- **Project path is passed via `OPEN_EDIT_PROJECT` env var.** The
  extension reads it, the bridge resolves it to a real folder.

### Files added

- `open_edit/serve/pi_bridge.py` — Python CLI (--tool --project --args)
  that dispatches to the 10 real tools + the virtual `trigger_render`.

- `open_edit/serve/pi_extension/extension.ts` — TypeScript pi extension.
  Loads the 11 tool schemas from Python, registers them with pi, and

### How to use

```bash
# 1. Install pi (already on this system via npm)
pi --version  # 0.80.10+

# 2. Start the server with the pi provider
export OPEN_EDIT_LLM_PROVIDER=pi
export OPEN_EDIT_LLM_MODEL=minimax-m3
export OPEN_EDIT_PI_PROVIDER=opencode-go
export OPEN_EDIT_PROJECTS_ROOT=~/OpenEditProjects
unset OPEN_EDIT_LLM_API_KEY       # not needed for pi

open_edit serve --port 8000
# → http://localhost:8000 (chat UI)
# → "add a marker at 2.5s saying hello"
# → LLM calls add_marker → pi extension runs it → bridge persists it
```

### Known limitations

- The `add_marker` schema was renamed from `timestamp` to `t_start`/`t_end`
  to match the real tool. If the LLM still uses `timestamp`, the tool
  will fail with a schema error. Tested with `minimax-m3`; works correctly.
- System prompt grows on every turn. After ~10 turns it may hit the
  context window limit. Future: cache the project state in a file
  the extension can `@`-reference.
- The `agent_end` event from pi is not specially handled; we treat
  every `done` as a normal turn end. If pi emits multiple turns per
  `--print` (e.g., with auto-retry), we may double-emit. Future: track
  turn boundaries explicitly.
